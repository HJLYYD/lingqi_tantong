# 灵柒·探瞳 (LingQi TanTong) C语言实现 — 项目全面分析

---

## 1. 项目目的

**灵柒·探瞳** 是一款面向 **国产 RISC-V 芯片（进迭时空 K1 / SpacemiT X60）** 的**智能探测箭系统**。系统分为两个物理端：

| 端 | 角色 | 硬件平台 |
|---|---|---|
| **箭矢端** | 感知与传输前端（摄像头 + IMU 传感器数据采集） | ESP32-P4 |
| **射手端** | 核心计算与 AR 显示单元（AI推理 + 目标跟踪 + 3D定位 + 可视化） | 进迭时空 K1 Muse Pi Pro (RISC-V) |

**核心功能**：对输入视频流进行**YOLOv8n行人检测、YOLOv8-Pose姿态估计、SCRFD人脸检测、ArcFace人脸识别、ByteTrack多目标跟踪、3D空间定位（针孔相机模型）、可视化渲染**，最终生成带标注的结果视频和分析报告（JSON/CSV）。

本仓库是射手端核心算法的 **C 语言高性能重写**，与上游 Python 原型保持功能对等，面向资源受限的 RISC-V 嵌入式平台。

## 4. 模块关系与数据流

### 分层架构

```
Application Layer:   main.c
                         │
Controller Layer:    system_controller.c  (10步帧处理流水线)
                         │
Business Logic:      inference_pipeline  │  tracking_manager  │  spatial_engine
                         │                      │                    │
Data Processing:     video_processor  │  imu_handler  │  model_store  │  arrow_receiver
                         │
Presentation:        visualizer  │  ar_renderer  │  video_writer
                         │
Support:             config_manager  │  logger  │  result_manager  │  utils
```

### 每帧处理流程（system_controller_process_video）

```
1. video_processor_read_frame_raw()          -- 读取原始帧数据
2. inference_pipeline_process_frame()        -- 级联 AI 推理 (YOLOv8n→Pose→SCRFD→ArcFace)
3. spatial_engine_initialize_coordinate_system() -- 首帧初始化世界坐标系
4. spatial_engine_calculate_position()       -- 2D bbox → 3D 空间坐标（针孔模型）
5. object_tracker_update()                   -- ByteTrack 多目标跟踪（三阶段匹配+Kalman）
6. associate_poses_with_objects()            -- IoU 关联姿态到跟踪目标
7. associate_faces_with_objects()            -- IoU 关联人脸到跟踪目标
8. spatial_engine_update_trajectory()        -- 更新轨迹历史
9. imu_handler → spatial_engine_set_camera_pose() -- 更新相机姿态
10. visualizer_render_detection_view()       -- 可视化渲染
    result_manager_update_session_stats()    -- 记录统计数据
```

### 关键模块依赖

- **所有模块** 依赖 `core_types.h`（共享数据结构）
- **所有模块** 依赖 `logger.h`（日志基础设施）
- **system_controller** 聚合所有子模块
- **inference_pipeline** 聚合 4 个 AI 模型模块（yolov8_detector, yolov8_pose_estimator, scrfd_detector, arcface_recognizer）
- **tracking_manager** 内置 KalmanBoxTracker（7状态 Kalman 滤波）
- **spatial_engine** 被 system_controller 直接调用，同时也消费 IMU 姿态数据

## 5. 硬件/架构假设

### 目标平台
- **主开发/部署目标**：进迭时空 K1 Muse Pi Pro（SpacemiT X60 RISC-V 64位处理器）
- **箭矢端**：ESP32-P4（RISC-V 32位）搭载 GY-87 传感器模组（MPU6050 + QMC5883L + BMP180）和摄像头

### RISC-V 向量化
- CMakeLists.txt 中 RISC-V 交叉编译标志 `-march=rv64gcv` 保留了对 RVV 0.7.1 向量扩展的支持（目前标记为规划中）

### ONNX Runtime 条件编译
- 通过 `#ifdef HAS_ONNX_RUNTIME` 实现双路径：
  - **有 ONNX Runtime**：真实模型推理
  - **无 ONNX Runtime**：纯 C 启发式算法回退（肤色检测、对称性分析等），确保无外部依赖时仍可构建运行

### 内存约束
- 固定上限设计：Detection <= 100, TrackedObject <= 256, Trajectory <= 300, Face <= 20
- 无全局变量，所有状态封装在模块 struct 中
- 显式 create/destroy 生命周期管理

### 通信协议
- 箭矢端 -> 射手端：自定义 Arrow 协议通过 UART（3Mbps 波特率），含 JPEG 图像帧和 IMU 姿态帧，CRC16-CCITT 校验
- 帧格式：`magic_start(0xA5 0x5A) + type + seq + timestamp + length + payload + crc16 + magic_end(0x5A 0xA5)`

### 坐标系
- 双坐标系设计：2D 像素坐标（快速显示） + 3D 世界坐标系（空间分析）
- 世界坐标系原点由首帧第一个检测目标自动标定

- **已实现**：YOLOv8n 检测、YOLOv8-Pose 姿态、SCRFD 人脸检测、ArcFace 识别、ByteTrack 跟踪、针孔相机模型定位、可视化渲染、配置管理、日志、结果管理、Mahony 滤波、KCP-Lite 协议、Arrow 协议

## 第三章：🔴 核心缺失模块 (阻塞实机部署)

### 3.1 ONNX Runtime RISC-V 交叉编译与集成

**当前状态**：`#ifdef HAS_ONNX_RUNTIME` 双路径代码已就绪，x86 预编译包已测试通过。但 **RISC-V 平台上不存在官方预编译 ONNX Runtime**。

**阻塞效应**：无 ONNX Runtime → 所有 4 个 AI 模型回退到**启发式算法** → 精度灾难性下降 (mAP < 10%) → 系统不可用。

**需要做的事情**：
```
1. 源码编译 ONNX Runtime v1.18+ for riscv64
   - cmake -DCMAKE_TOOLCHAIN_FILE=cmake/riscv64-toolchain.cmake
   - -DONNX_RUNTIME_USE_XNNPACK=ON (使用 XNNPACK RISC-V 后端)
   - -DCMAKE_BUILD_TYPE=MinSizeRel (K1 内存受限)
   
2. 关键挑战：
   - XNNPACK 对 RVV 0.7.1 的支持成熟度 (当前偏 RVV 1.0)
   - ORT 依赖 protobuf/abseil 的 RISC-V 交叉编译
   - 建议：先在 Bianbu 实机上原生编译，跳过交叉编译复杂度
```

### 3.2 Spacengine NPU INT8 推理适配

**当前状态**：**完全未集成**。所有推理路径假设 CPU 执行。

**阻塞效应**：K1 NPU 是唯一能让 AI 推理达到 **30 FPS** 的路径。纯 CPU 推理预计 **1-3 FPS**，远不满足实时要求。

**需要做的事情**：
```
1. 模型量化管线
   - ONNX fp32 → TensorFlow Lite INT8 → Spacengine NPU
   - 需要 500+ 张校准图片
   - YOLOv8n: mAP 损失 <3% 目标
   - YOLOv8-Pose: AP 损失 <5% 目标
   - SCRFD: AP 保持 >90%
   - ArcFace: TAR@FAR 保持 >95%

2. NPU 推理 API 适配层
   src/inference/npu_adapter.c/h
   - spacengine_load_model() 替换 ort_create_session()
   - spacengine_run() 替换 ort_run()
   - 输入/输出 tensor 格式转换 (NHWC ↔ NCHW)

3. 内存管理优化
   - NPU 通常需要连续物理内存 (ION/DMA-BUF)
   - 模型权重常驻 NPU SRAM，不要每帧重新加载
```

### 3.3 VINS-Mono 视觉-惯性里程计

**当前状态**：🔴 完全缺失。仅依赖**单帧针孔模型 + 恒定平均身高先验**进行深度估计，误差随距离增大指数增长（>5m 时 Z 误差可达 50%+）。

**重要性**：VINS-Mono 是 6-DOF 精确位姿估计的唯一路径，直接影响：
- 目标 3D 坐标精度
- AR 叠加稳定性
- ATW 姿态预测精度

**需要实现**：
```
src/vins/
├── imu_preintegrator.c   — IMU 预积分 (加速度/角速度 → 帧间相对位姿)
├── marginalization.c     — Schur 补边缘化先验
├── feature_tracker.c     — KLT 光流特征追踪 (RISC-V 优化)
├── estimator.c           — 滑动窗口 Bundle Adjustment (8 帧窗口)
```

**Bianbu 适配要点**：
- Ceres Solver 需交叉编译 riscv64 版本
- 滑动窗口 BA 涉及大量线性代数 → 考虑使用 Eigen RISC-V 后端

### 3.4 ATW 异步时间扭曲 (Motion-to-Photon ≤ 17.8ms)

**当前状态**：`ar_renderer_compensate_motion()` 仅做逐像素 Euler 旋转（O(W×H) 遍历），**既慢又不对齐 GPU 渲染管线**。此代码路径在 K1 上预期耗时 **80-200ms/frame**，远超 17.8ms 目标。

**需要实现**：
```

   - 顶点着色器：IMU 姿态 → MVP 矩阵
   - 片段着色器：异步时间扭曲纹理采样
   - FBO (帧缓冲对象) 离屏渲染

2. IMU 线程与渲染线程解耦
   - IMU 线程：200Hz 采集 → 四元数队列 (lock-free ring buffer)
   - 渲染线程：读取最新 IMU 四元数 → 姿态外推 Δt → 扭曲矩阵

3. GPU 纹理映射管线
   - 等距投影 → 旋转 → 逆投影
   - VSync 同步：glFinish() 强制 GPU 完成

目标：MTP (Motion-to-Photon) ≤ 17.8ms (人眼感知阈值 20ms)
```

## 第四章：🟠 性能瓶颈深度分析

### 4.1 推理管线 — 级联开销

[inference_pipeline.c](file:///d:/shool/大三下/embedded/lingqi_tantong_c/src/inference_pipeline.c#L246-L278) 每帧执行 **4 次 ONNX 推理**级联：

```
YOLOv8n (640×640)  →  YOLOv8-Pose (每个Person crop)  →  SCRFD (640×640)  →  ArcFace (112×112)
```

**K1 上的预期延迟**：
| 模型 | CPU FP32 | CPU INT8 (XNNPACK) | NPU INT8 (目标) |
|------|----------|---------------------|------------------|
| YOLOv8n | ~800ms | ~400ms | **~15ms** |
| YOLOv8-Pose (per person) | ~300ms | ~150ms | **~8ms** |
| SCRFD | ~500ms | ~250ms | **~12ms** |
| ArcFace | ~120ms | ~60ms | **~5ms** |
| **总计** | **~1720ms** | **~860ms** | **~40ms** (25fps!) |

**结论**：没有 NPU，整个推理管线在 K1 上约为 **0.5-1.7 秒/帧**，完全不可用。

### 4.2 Kalman 滤波器 — 矩阵密集运算

[tracking_manager.c](file:///d:/shool/大三下/embedded/lingqi_tantong_c/src/tracking_manager.c#L64-L162) 中 Kalman predict/update 每帧执行 N_tracks 次，每次涉及：
- 7×7 矩阵乘法 (49 次循环 × 7 FLOPs = 343 FLOPs)
- 4×4 矩阵求逆 (`utils_matrix_inverse_4x4`, [utils.c](file:///d:/shool/大三下/embedded/lingqi_tantong_c/src/utils.c#L134-L181)，Gauss-Jordan 消元 ~64 FLOPs)
- 多个 7×4 和 4×7 矩阵乘法

**预计 K1 上每个 Kalman 更新**: ~15-25μs (RISC-V 无 SIMD)
**x86 对比**: ~2-3μs (AVX2 自动向量化)

**优化方向**：
1. 使用 `__attribute__((aligned(16)))` 对齐 `float[7][7]`
2. 手动展开 7×7 矩阵乘法内循环
3. 将 Kalman 协方差矩阵改为对角存储（精度换速度）

### 4.3 可视化渲染 — 像素级 CPU 遍历

[visualizer.c](file:///d:/shool/大三下/embedded/lingqi_tantong_c/src/visualizer.c) 所有渲染在 CPU 上逐像素完成：
- `draw_rect`: O(W+H) 嵌套循环，每像素带边界检查
- `draw_line`: Bresenham 算法 + thickness 嵌套
- `draw_circle`: O(R²) 距离计算
- `draw_text`: 5×7 bitmap 逐 bit 遍历
- 整个 `visualizer_render_detection_view` 包含 `memcpy(full_frame)` → N 个 draw_rect → N 个 draw_label → N 个 draw_skeleton

**K1 上一帧 1920×1080 可视化**: ~40-80ms (RISC-V 单核)

**优化方向**：
- 将可视化渲染整个搬迁到 GPU (OpenGL ES 3.2)
- CPU 端仅做坐标计算，不再做像素渲染
- 或使用 RISC-V RVV intrinsics 加速 `memcpy` 和像素填充

### 4.4 AR 运动补偿 — O(W×H) 逐像素变换

[ar_renderer.c](file:///d:/shool/大三下/embedded/lingqi_tantong_c/src/ar_renderer.c#L96-L163) 中 `ar_renderer_compensate_motion` 对每个像素执行 12 次三角函数 + 3D 旋转矩阵：

```c
// 1920×1080 = 2,073,600 像素
for (int y = 0; y < height; y++)       // 1080 次
    for (int x = 0; x < width; x++)    // 1920 次
        // cos/sin × 6 + 矩阵 × 12 = 18 次浮点运算
        // 总计: ~37M FLOP + 2M 次 cos/sin 调用
```

**K1 上单帧**: 预期 **200-500ms** (完全不可接受)。

**必须搬迁到 GPU**：片段着色器对 UV 坐标进行单次矩阵变换，GPU 并行处理全部像素。

### 4.5 Letterbox / 图像预处理 — 可向量化的瓶颈

[utils.c](file:///d:/shool/大三下/embedded/lingqi_tantong_c/src/utils.c#L57-L100) 中的 `utils_letterbox` 和 `utils_normalize_chw` 是 YOLOv8n 推理前的关键预处理，涉及 O(W×H) 像素遍历。RISC-V RVV 0.7.1 (256-bit VLEN) 可以一次处理 8 个 float32 像素，理论上 **8× 加速**。

## 第九章：🎯 总结 — 距离落地还有多远？

### 已完成 (✅)
- **射手端核心算法 C 代码**：完整、结构清晰、模块化良好
- **ESP32-P4 箭矢端固件**：FreeRTOS 三任务 + GY87 + 协议
- **KCP-Lite / Arrow / Mahony**：三大通信/姿态模块已实现
- **双路径设计**：ONNX/启发式，确保无外部依赖仍可构建

### 核心不足 (❌)
1. **NPU 推理完全空白** — 这是 K1 上达到 25+ FPS 的唯一路径
2. **VINS-Mono 视觉-惯性里程计缺失** — 3D 定位精度无法满足生产要求
3. **ATW 异步时间扭曲未嵌入 GPU 管线** — MTP >80ms，AR 体验不可接受
4. **MiDaS 深度估计未实现** — 远距精度崩溃
5. **RISC-V 交叉编译未验证** — 从未在实机上编译运行
6. **实时视频流 (V4L2/UART) 通路未打通** — 仅支持离线 MP4
7. **GPU 渲染管线完全空白** — 所有可视化在 CPU 上

### 乐观估计
在 NPU SDK 可用的前提下，**6-8 周**可完成 Phase 0-1 的端到端通路打通（15fps 基本可用）。完整体验级交付（30fps + AR + 动作识别）仍需 **14-20 周**。

### 最大不确定性
**SpacemiT Spacengine NPU SDK 的成熟度和可用性**。这是整个项目能否在 K1 上实时运行的决定性因素。建议尽早联系进迭时空获取 SDK 并进行基准测试。如果 NPU 不可用，需要考虑备选方案（如使用更轻量的 NanoDet-Plus 替换 YOLOv8n，或将推理卸载回 ESP32-P4 的 NPU）。
