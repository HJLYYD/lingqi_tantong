# 问题分析报告 — 灵契探瞳目标追踪系统

> 文档版本: v2.0  
> 日期: 2026-06-08  
> 分析对象: `lingqi_tantong_c` 目标追踪系统 (SpacemiT K1 / Muse Pi Pro 平台)

---

## 一、执行摘要

本报告对灵契探瞳目标追踪系统进行了全面的代码审查和运行时行为分析，定位了三个关键功能性缺陷的根本技术原因：

1. **追踪稳定性问题**（ID跳变、追踪中断）：根因在于纯贪婪IoU匹配 + 固定阈值 + 无外观特征
2. **多人同时识别失效**：根因在于级联状态机过早优化 + 关键点验证器对远/小目标过于严格
3. **遮挡场景识别不足**：根因在于无部分身体检测 + 丢失轨迹处理幼稚 + 无重识别机制

---

## 二、系统架构概览

### 2.1 当前追踪管线

```
摄像头/YOLO检测 → 空间位置计算 → 目标追踪 → 姿态关联 → 可视化输出
                     ↑                ↑           ↑
               SpatialEngine   TrackingManager  PoseEstimation
```

### 2.2 核心模块

| 模块 | 文件 | 功能 |
|------|------|------|
| 推理管线 | `inference_pipeline.c` | YOLOv8-Pose(主) + YOLO11n(副) + 人脸 + 动作识别 |
| 追踪管理器 | `tracking_manager.c` | ByteTrack风格两阶段匹配 + 7状态卡尔曼滤波 |
| 关键点验证 | `keypoint_validator.c` | 解剖学合理性检查（肩对称、躯干比例、四肢比例） |
| 系统控制器 | `system_controller.c` | 管线编排、帧循环、结果输出 |

## 三、问题一：追踪稳定性问题

### 3.1 问题表现

- 同一人在运动过程中追踪ID频繁跳变（如 Person_1 → Person_5 → Person_12）
- 两人交叉行走后ID互换
- 短暂遮挡后追踪完全丢失，重新出现时分配新ID

### 3.2 根因分析

#### 根因1：纯贪婪IoU匹配 (`tracking_manager.c:431-455`)

```c
// 当前代码：贪婪最佳匹配（局部最优，非全局最优）
for (int t = 0; t < num_predicted; t++) {
    float best_iou = TRACKING_IOU_THRESHOLD;
    int best_det = -1;
    for (int h = 0; h < num_high; h++) {
        float iou = bbox_iou(&predicted_bboxes[t], &detections[d].bbox);
        if (iou > best_iou) { best_iou = iou; best_det = d; }
    }
    // ... 匹配第一个满足阈值的检测
}
```

**问题**：贪婪算法按轨迹顺序匹配，先到先得。当两个轨迹的预测框与同一检测都有高IoU时，第一个轨迹"抢走"检测，第二个轨迹只能匹配次优检测或丢失。这是局部最优而非全局最优，导致ID交换。

#### 根因2：无外观特征 (`tracking_manager.c` — 全局)

当前系统**完全依赖IoU**进行数据关联。没有任何形式的外观特征（ReID嵌入、颜色直方图、关键点描述符）。当IoU因运动模糊、遮挡或快速移动而下降时，关联完全失败。

#### 根因3：固定IoU阈值 (`tracking_manager.h:24`)

```c
#define TRACKING_IOU_THRESHOLD 0.35f  // 所有场景统一阈值
```

快速移动时卡尔曼预测框与实际检测框的IoU可能低于0.35，导致匹配失败。遮挡场景下可见部分IoU更低。

#### 根因4：无级联匹配

系统将所有轨迹（不论丢失多久）与所有检测混合匹配。丢失30帧的陈旧轨迹与刚更新1帧的活跃轨迹平等竞争——而陈旧轨迹的卡尔曼不确定性更大（协方差矩阵发散），Mahalanobis距离反而更小，更容易"抢到"检测。

### 3.3 影响量化

| 指标 | 优化前估计值 | 目标值 |
|------|------------|--------|
| ID保持率（30分钟） | ~60-70% | ≥99% |
| ID交换次数/分钟 | ~3-5次 | <0.1次 |
| 交叉行走ID互换率 | ~80% | <5% |

## 四、问题二：多人同时识别失效

### 4.1 问题表现

- 场景中仅有1人被稳定追踪
- 第2人进入画面后，需等待数秒到数十秒才被检测
- 3人以上同时出现时，只有1-2人被追踪

### 4.2 根因分析

#### 根因1：级联状态机过早优化 (`inference_pipeline.c:517-566`)

```c
case PIPELINE_CASCADE_TRACKING:
    // 当≥1个确认轨迹存在时 → 完全跳过YOLO11n
    // 仅在每15帧的VALIDATING帧运行完整检测
```

**问题**：一旦有1人被确认追踪，系统立即进入TRACKING模式，**完全跳过YOLO11n副检测器**。新人进入场景后，只能依赖YOLOv8-Pose的检测框——而Pose模型的DFL信号本身就不可靠（INT8量化破坏了cls/obj头）。新人需等待最多15帧（约0.5-1秒）才能被VALIDATING帧捕获。

#### 根因2：关键点验证器过于严格 (`keypoint_validator.c:338-361`)

```c
// quick_check 要求至少一组配对肢体可见（双肩、双髋或双膝）
if (!has_shoulders && !has_hips && !has_knees) {
    return false;  // 拒绝检测
}
```

对于远处的人（小边界框，关键点置信度低）或部分可见的人，此条件难以满足。

#### 根因3：NMS链可能合并邻近人物

```
检测器内部NMS 0.30 → 合并IoU 0.35 → 最终NMS 0.35
```

三个NMS阶段均在0.30-0.35范围内，两个靠近站立的人可能被合并为一个检测。

### 4.3 影响量化

| 指标 | 优化前估计值 | 目标值 |
|------|------------|--------|
| 最大同时追踪人数 | ~2人 | ≥10人 |
| 新人检测延迟 | 0.5-1秒 | <0.2秒 |
| 多人准确率 | ~60% | ≥95% |

## 五、问题三：遮挡场景识别不足

### 5.1 问题表现

- 人物下半身被桌子/围栏遮挡时，完全无法识别
- 侧身行走（仅一侧关键点可见）时检测丢失
- 70%遮挡情况下追踪完全中断

### 5.2 根因分析

#### 根因1：无部分身体检测

系统仅使用全身检测器（YOLOv8-Pose + YOLO11n），两者都期望完整的全身。没有上半身专用检测器或侧身检测器。关键点验证器的`quick_check`要求至少一组配对肢体（双肩或双髋或双膝）——当这些都被遮挡时，验证失败，检测被丢弃。

#### 根因2：丢失轨迹处理幼稚 (`tracking_manager.c:487-496`)

```c
// 未匹配的轨迹仅进行卡尔曼预测 + 置信度衰减
tracker->tracks[track_idx].object.detection.confidence *= 0.9f;
```

无外观特征匹配、无轨迹复活机制、无重识别。丢失30帧后永久删除。

#### 根因3：无遮挡感知匹配

当前匹配策略不区分"轨迹因遮挡短暂丢失"和"轨迹因离开画面而丢失"。两种情况的处理完全相同。

#### 根因4：关键点验证拒绝部分身体

当人体被遮挡（如下半身），关键点验证要求双肩+双髋+双膝至少一组配对可见。遮挡场景中这些条件都可能不满足。

### 5.3 影响量化

| 指标 | 优化前估计值 | 目标值 |
|------|------------|--------|
| 30%遮挡识别率 | ~40% | ≥85% |
| 50%遮挡识别率 | ~10% | ≥85% |
| 70%遮挡识别率 | ~0% | ≥85% |
| 上半身识别率 | ~30% | ≥85% |
| 侧半身识别率 | ~15% | ≥85% |
